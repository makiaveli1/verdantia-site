# No-fail presentation protocol

[DRAFT — INTERNAL OPERATING STANDARD]

Reliability is part of the Verdantia offer. Every paid briefing, workshop, adoption day, or sprint session should work even if Wi-Fi, live demos, or devices misbehave.

## Rule

Never rely on a live demo as the only path to teach the point.

## 1. One week before

- Confirm audience size, room format, timing, tool access, and participant device expectations.
- Confirm whether participants can use external AI tools during the session.
- Ask about blocked websites, VPN, guest Wi-Fi, Teams/Zoom policies, and screen-sharing constraints.
- Confirm that live exercises will use synthetic, anonymised, or explicitly approved data only.
- Confirm whether client/customer/employee data may ever be used in approved internal tools; default is no.
- Confirm accessibility needs.
- Confirm whether recording is allowed.

## 2. Two days before

- Export the deck to PDF.
- Save a local/offline copy of the deck.
- Prepare participant handout/checklist as PDF.
- Prepare screenshots or short screen recordings for any demo that could fail.
- Prepare a simple text-only fallback version of each exercise.
- Send any pre-work or tool setup instructions.

## 3. Day before

- Charge laptop and backup device if available.
- Download all required assets locally.
- Open all required browser tabs and documents.
- Test links.
- Test HDMI/USB-C adapter if presenting in person.
- Prepare hotspot/backup connection where possible.
- Put deck, PDFs, screenshots, and worksheets in one clearly named folder.

## 4. 30–60 minutes before

- Join early or arrive early.
- Test audio, screen share/projector, browser, and AI tools.
- Open deck in both editable format and PDF.
- Open backup screenshots/video.
- Close private/unrelated tabs and notifications.
- Confirm session timing with organiser.

## 5. During the session

- Start with outcomes and structure.
- Name the fallback calmly if technology fails.
- Do not troubleshoot live for more than 2–3 minutes unless the session depends on it.
- If a live AI tool fails, teach from screenshots, examples, and the framework.
- Stop any exercise that risks exposing personal, client, confidential, or commercially sensitive data.
- Keep exercises simple enough to complete without perfect tooling.
- Capture useful questions and use cases.

## 6. After the session

- Send promised materials.
- Send a short summary of use cases, risks, and recommended next steps.
- Note what worked, what confused participants, and what should change in the next delivery.
- If appropriate, propose the next offer: Adoption Day or Adoption Sprint.

## Emergency fallback script

> The tool is not cooperating, which is annoying but useful. This is exactly why AI adoption needs process, not just demos. I’ll show the same workflow using the prepared example, then we’ll apply the checklist so you still leave with the method.

## Minimum kit

- Laptop charger.
- HDMI/USB-C adapter.
- PDF deck.
- Offline deck.
- Screenshots/video backups.
- Participant worksheet.
- Responsible-use checklist.
- Hotspot/backup internet where possible.
- Water. Because humans remain tragically non-rechargeable.
