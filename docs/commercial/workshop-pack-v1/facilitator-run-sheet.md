## Session setup

- Duration: 3–3.5 hours plus breaks
- Standard group size: up to 20 participants
- Delivery mode: remote / onsite / hybrid
- Tools: organisation-approved tools only
- Data rule: synthetic, anonymised, or approved examples only

## Timing plan

| Segment | Time | Purpose | Facilitator notes |
|---|---:|---|---|
| Opening and baseline | 15 min | Learn current confidence/use | Ask what people have tried and what worries them |
| Tool literacy | 30 min | Ground tool choice in work | Avoid vendor fandom; anchor on task fit |
| Prompting practice | 40 min | Teach reusable structure | Use weak prompt -> stronger prompt exercise |
| Role-based practice | 55 min | Apply to real work | Choose 2–4 lanes from scoping call |
| Safe-use habits | 30 min | Build review discipline | Emphasise data boundaries and human accountability |
| Use-case capture | 25 min | Identify next experiments | Rank by value/risk/ease and owner |

Core facilitation time: **195 min / 3h15**. Add a 10–15 minute break and keep transitions tight if the booked slot is 3.5 hours.

## Opening script

> The goal today is not to chase every AI tool or collect a bag of prompts. The goal is to build better judgement: where AI can help your work, where it can mislead you, and what needs human review before anything is used.

## Tool literacy framing

Use this line when the group asks “which AI is best?”:

> The better question is: what work are we improving, what data is involved, what level of risk exists, and what review is needed?

## Prompting frame

Role. Task. Context. Constraints. Output format. Review criteria.

Repeat this until it becomes boring. Boring is how useful habits are born. Annoying but true.

## Safety interruption line

If a participant is about to use sensitive data:

> Let’s pause there. For this workshop, we’ll use a synthetic or anonymised version of that example. The habit we want is useful AI practice without exposing real client, employee, or confidential data.

## Demo failure fallback

> The tool is not cooperating, which is annoying but useful. This is exactly why AI adoption needs process, not just demos. I’ll show the same workflow using the prepared example, then we’ll apply the checklist so you still leave with the method.

## Closing script

> The win after today is not that everyone uses AI for everything. The win is that the team can spot sensible use cases, avoid obvious risks, prompt with better context, and review outputs before relying on them.

## After-session notes

Record immediately after delivery:

- What examples landed well?
- What confused participants?
- What risks came up?
- What use cases were strongest?
- Who seemed to own follow-up?
- What should change in the next version?
